## What We Will Validate

This eval is **source-inspection-dominant** because runtime GUI testing was deliberately skipped to protect tester's live skill environment (see claim-009 skip_reason).

- ✅ claim-001: Every platform declared in README also seeded in `builtin_agents()`
- ✅ claim-002: Symlink-based install is real (`std::os::unix::fs::symlink`), not copy
- ✅ claim-003: `is_agent_detected()` reflects real filesystem, not cache
- ✅ claim-004: GitHub import retry/mirror fallback is real code path
- ✅ claim-005: DB upserts are idempotent (`ON CONFLICT DO UPDATE`)
- ✅ claim-006: No telemetry libraries; SQLite path exact
- ✅ claim-007: Prebuilt DMG integrity (SHA256, bundle, signing status)
- ✅ claim-008: Bidirectional centralize works (code-level)
- ⏭️ claim-009: Live GUI workflow — deliberately skipped

## Real Inputs We Will Use

- `repos/iamzhihuix--skills-manage` shallow clone from `main` at the time of eval
- GitHub Release v0.9.1 DMG asset (`skills-manage_0.9.1_macos_universal.dmg`)
- Tester's live filesystem: `ls ~/.agents/`, `ls ~/.claude/skills/`, `ls ~/.skillsmanage/` (non-existent)

## How Many Times We Will Test

- Source inspection: once, grep + read, with line refs
- DMG integrity: once (SHA256 + mount + codesign)
- Runtime workflow: **zero** (see skip)

## What Counts As Passing

- Every README platform row maps to a `builtin_agents()` entry AND directory path matches
- `create_symlink` uses real OS symlink syscalls, not copy
- `is_agent_detected` returns false when both dir and parent missing
- GitHub import has ≥2 mirror fallbacks
- `upsert_skill_installation` query uses `ON CONFLICT`
- Zero analytics/telemetry deps in `package.json` and `src-tauri/Cargo.toml`
- DMG SHA256 = published digest; bundle has valid Mach-O universal binary
- `ensure_centralized()` copies skill into central before install

## If Everything Passes, What We Can Trust

- **Minimum trust level: `usable`** — code path for every claim exists and is reviewable
- **Stretch trust level: `reusable`** — but only after a live runtime workflow test (claim-009) in a clean environment
- **Not `recommendable`** — because v0.9.1 is 11 days old, adhoc-signed, has 3 self-disclosed failing frontend tests, and we haven't proven the end-to-end workflow at runtime

## Known Deviations & Risks

- README → code inconsistencies (all minor):
  - README: React 19 — actual: `react ^18.3.1`
  - README: "Apple Silicon macOS" — actual: universal binary (under-claim, favorable)
  - README: Hermes in `Coding` category — code: `lobster` category
  - README: EasyClaw V2 listed — no matching `id` seeded in code
- CLAUDE.md self-discloses 3 legacy failing frontend tests
- API keys stored unencrypted in SQLite (README discloses this honestly)
- For users with heavy pre-existing skill state across multiple tools (dbskill, lobster lock file, etc.), introducing another source-of-truth layer is a real conflict risk
