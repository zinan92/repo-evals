# RedBox eval plan — 2026-05-04

> Archetype: **orchestrator**. Layer: **compound** (RedClaw automation
> console runs LLM-driven multi-step tasks; background scheduler keeps
> long-running work alive). Per the framework, compound eval has a hard
> ceiling at `usable` until at least one fully logged scenario exists,
> and `reusable` until at least three.

## What this repo claims

Pulled from the English README, browser-extension manifest, and desktop
package.json (links in `claims/claim-map.yaml`):

- 跨平台桌面 AI 工作台，签名安装包覆盖 macOS / Windows / Linux
- 浏览器插件 (manifest v3) 一键抓取小红书、抖音、Bilibili、快手、TikTok、
  Reddit、X、Instagram、YouTube
- 桌面端基于 Electron 39 + Vercel `ai` SDK，支持自定义 endpoint / API key /
  model（Anthropic / OpenAI / Google）
- 端到端创作流程：抓取 → 知识库 → AI 编辑器 → 配图
- RedClaw 自动化控制台单 session 内独立完成任务
- 后台调度任务长跑，关掉窗口照常运行

## What we will validate now (static, in-repo)

| Claim | What we check | Method |
|---|---|---|
| claim-001 distribution | release assets exist for all 3 OS, sizes look real | `gh api repos/Jamailar/RedBox/releases` |
| claim-002 capture coverage | manifest host_permissions ⊇ README platform claims | fetch manifest.json, diff against README list |
| claim-003 ai providers | package.json carries Vercel ai SDK + 3 providers | fetch desktop/package.json |

## What we cannot validate now (compound, scenario-driven)

These require the user to install the binary, configure an
endpoint/key/model, and exercise the workspace by hand. The dashboard
renders these as compound-experiment scenarios with system prompts and
watch-for criteria:

- claim-004 end-to-end creation flow
- claim-005 RedClaw single-session autonomy
- claim-006 background scheduling
- claim-007 user-friendly failure modes

## Real inputs we will use

- claim-001: gh api releases, http HEAD on each asset
- claim-002: manifest.json from `main` branch (commit-pinned in evidence)
- claim-003: desktop/package.json from `main` branch
- claims-004..007: real user session on RedBox v1.11.0 — captured under
  `runs/<date>/run-<scenario>/business-notes.md` after the user runs it

## How many times we will test

- Static claims: 1 pass on the current `main` snapshot
- Compound claims: ≥1 logged scenario per claim before promoting beyond
  `usable`; ≥3 before considering `reusable`

## What counts as passing

- claim-001: every README-listed installer URL returns 200 with size > 50 MB
- claim-002: every README platform appears in manifest host_permissions
  (with the standard www. and m./mobile. variants where stated)
- claim-003: `@ai-sdk/anthropic`, `@ai-sdk/openai`, `@google/genai`,
  and `ai` v6 all present in dependencies; no commercial-only proxy
  hardcoded
- claim-004..007: see compound-experiment scenarios; each needs a real
  artefact + observed call graph + verdict (pass / pass-with-concerns /
  fail / untested) logged in business-notes.md

## If everything passes, what we can trust

- Static claims passing → `usable`: distribution and providers are real.
- Plus 1 compound scenario covering claim-004 → `usable` confirmed,
  remains capped because RedClaw + scheduler still untested.
- All four compound claims with logged passing scenarios → `reusable`.
- Multi-evaluator + multi-OS coverage → `recommendable`.

Static claims alone cannot promote past `usable` per the compound layer
ceiling rule.
