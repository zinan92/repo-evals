# Eval Plan

## What This Repo Claims

- It can serve as a single entrypoint for multiple content-related capabilities.
- It can route user intent to the correct downstream capability.
- It can orchestrate multi-step content workflows without the user needing to know internal tool boundaries.
- It can make underlying capabilities easier to use as one coherent system.

## What We Will Validate

- Orchestration: does the toolkit send the user to the right capability for a clear request?
- Workflow completion: can a realistic content task finish end to end through the toolkit?
- Delegation quality: when the toolkit calls a downstream capability, is the result still usable?
- Boundary handling: if a request is unclear or unsupported, does the toolkit fail clearly?

## Real Inputs We Will Use

- Simple single-step requests
- Multi-step content workflow requests
- Requests that should route to downloader / extractor / rewriter
- Ambiguous or unsupported prompts

## How Many Times We Will Test

- Core path: at least 3 representative tasks
- Variation path: at least 1 scenario per major downstream area
- Repeatability check: repeat the same task 3 times for at least one important workflow

## What Counts As Passing

- The repo completes the intended task
- The output is usable for the intended business goal
- Repeated runs stay within an acceptable quality band
- Failures are clear when the repo cannot complete the task

For this repo specifically:

- The toolkit chooses the right downstream capability often enough to be trusted
- The orchestration layer does not make the user experience worse than using the capability directly
- The user can stay at the toolkit layer without needing to understand internal repo boundaries

## If Everything Passes, What We Can Trust

- Minimum trust level: usable
- Stretch trust level: reusable
- Remaining risk: downstream capability quality may still limit the overall verdict
