# Eval Plan — park-intel

## What This Repo Claims

- `claim-001` 核心无密钥源可直接采集并进入健康状态
- `claim-002` 核心 API 真实返回结构化数据
- `claim-003` 前端 build 后由后端一起提供
- `claim-004` 可选源在无凭据时优雅降级
- `claim-005` 测试套件处于健康状态
- `claim-006` 详细健康视图语义一致且可解释
- `claim-007` launchd 背景服务脚本可用
- `claim-008` 事件聚合接口可用

## What We Will Validate

- 按 README quick start 安装 Python 依赖，确认服务具备启动前提
- 跑 `pytest tests/`，验证 README 的测试健康信号是否属实
- 启动真实服务，验证 `claim-001` 与 `claim-002`
- 补跑 `frontend` build，然后验证 `claim-003`
- 在无可选密钥的情况下观察 health 输出，验证 `claim-004`
- 对照 `/api/health` 与 `/api/health/sources`，检查 `claim-006`
- 请求 `/api/events/active`，至少确认事件接口 shape 对 `claim-008` 成立

## Real Inputs We Will Use

- Input A: fresh clone at local repo commit `3c6eaa9`
- Input B: default `.env.example` copied to `.env`，不额外填可选密钥
- Input C: live API on fallback port `8013`（因为本机 `8001` 已被占用）

## How Many Times We Will Test

- Python install: 1 次
- pytest: 1 次全量
- Server start: 至少 2 次
  第 1 次用 README 默认端口，观察是否可启动
  第 2 次用备用端口 `8013` 验证 live API
- API calls: 每条核心 endpoint 至少 1 次
- Frontend serving: `/` 和 `/health` 各 1 次

## What Counts As Passing

- `claim-001`: 多个核心 source 在无密钥条件下有真实数据且 health 可见
- `claim-002`: `/api/health`、`/api/articles/latest`、`/api/ui/feed` 返回非空结构化 JSON
- `claim-003`: frontend build 成功，`/` 和 `/health` 返回前端 HTML
- `claim-004`: 缺少可选凭据时，health 输出能解释 disabled/no_data
- `claim-005`: pytest 退出码 0；否则记为失败
- `claim-006`: health 总览与详细视图不应给出互相打架的 source 状态语义
- `claim-008`: `/api/events/active` 至少返回合法 JSON schema

## If Everything Passes, What We Can Trust

- 最低 trust level: `usable`
  代表服务能装、能起、能采、能回核心 API 和前端
- Stretch trust level: `reusable`
  代表测试健康、health 语义清楚、可选源降级路径也让人放心
- Remaining risk:
  launchd 脚本、长期运行稳定性、外部依赖波动、事件聚合质量
